## Maintainer Responsibilities

### Code Review
- Review pull requests in areas of expertise
- Provide constructive feedback
- Ensure code quality and consistency
- Test critical changes

### Project Management
- Triage issues and pull requests
- Prioritize work and milestones
- Facilitate discussions
- Mentor contributors

### Release Management
- Prepare releases
- Test release candidates
- Update documentation
- Publish releases

### Community Engagement
- Answer questions in Discussions
- Help new contributors
- Represent the project
- Gather feedback

## Becoming a Maintainer

### Requirements
To become a maintainer, you should:
1. **Consistent Contribution**
   - Regular contributions over at least 3 months
   - Multiple significant PRs merged

2. **Technical Expertise**
   - Deep understanding of project architecture
   - Quality code reviews
   - Good judgment on technical decisions

3. **Community Participation**
   - Active in discussions
   - Helpful to other contributors
   - Professional and respectful conduct

4. **Availability**
   - Time for reviews and discussions
   - Reasonable response time (within a week)
   - Advance notice for extended absences

### Process
1. Current maintainer nominates candidate
2. Candidate accepts nomination
3. Discussion period (minimum 2 weeks)
4. Approval by majority of maintainers
5. Added to MAINTAINERS.md and CODEOWNERS

### Stepping Down
Maintainers may step down at any time:
- Notify other maintainers in advance
- Document current work
- Help transition responsibilities
- Moved to emeritus status in recognition of contributions

## Maintainer Removal

Maintainers may be removed for:
- **Inactivity**: No contributions for 6+ months without notice
- **Conduct violations**: Repeated violations of Code of Conduct
- **Gross negligence**: Repeated harmful decisions or actions

Process:
1. Other maintainers discuss concerns
2. Maintainer in question given opportunity to respond
3. Vote by other maintainers (2/3 majority required)
4. Removal with explanation (private unless conduct violation)

## Access and Permissions

### GitHub Permissions
- **Write access**: All maintainers
- **Admin access**: Project lead and designated delegates
- **Protected branches**: Require maintainer approval

### Infrastructure Access
- **crates.io**: Designated maintainers
- **GitHub Pages**: Designated maintainers
- **CI/CD secrets**: Project lead and DevOps team

## Decision Making

### Consensus
- Decisions made through discussion and consensus
- Mainteners have veto power in areas of expertise
- Project lead breaks ties when needed

### Voting
Used when consensus cannot be reached:
- Simple majority for most decisions
- 2/3 majority for maintainership changes
- Project lead has veto power
